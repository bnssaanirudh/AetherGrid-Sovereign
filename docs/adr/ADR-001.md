# ADR-001: Monorepo and Package Boundaries

## Status
Proposed

## Context
AetherGrid-Sovereign has evolved from a fragile research script into an enterprise/industry continuation. The flat package structure causes circular imports and makes it difficult to maintain clean boundaries between core logic, data preparation, evaluation, and legacy components.

## Decision
We establish a monorepo structure under `packages/` with the following distinct libraries:
- `aethergrid_core`: Core framework-agnostic schemas, graph declarations, and validation helpers.
- `graph_builder`: Pipelines for loading real and synthetic spatial/outage graphs.
- `watchdog`: Quality, quarantine, and structural validation guards.
- `models`: HGT baseline, fuzzy/quantum operators, and future CV-PFA/VQC components.
- `sampling`: Temporal and ego network neighborhood sampling.
- `training`: Loss functions, datasets, trainers, and safely versioned checkpoint loaders.
- `evaluation`: Multi-task validation, metrics evaluation, and temporal split leakage detectors.
- `theory`: Mathematical verification of cascades and fuzzy boundaries.
- `aethergrid_legacy`: Archives of chaotic maps, Q-AVOA NAS, and old toy implementations.

## Consequences
- Imports must cross package boundaries explicitly.
- Legacy optimization code is isolated, removing distraction from the primary VQC-centric architecture.
